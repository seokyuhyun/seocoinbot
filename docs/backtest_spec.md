# 백테스트 명세서 v0.1

> 이 문서는 전략 규칙 v0.1(설계서 2절)을 과거 데이터로 검증하기 위한 명세다.
> Claude Code에 이 문서와 설계서를 함께 첨부하고 "이 명세대로 백테스트
> 코드를 만들어줘"라고 지시한다.

---

## 1. 백테스트의 목적

전략 규칙 v0.1이 **검증된 적 없는 가설**임을 전제한다.
이 백테스트의 목적은 "이 전략이 좋다"를 증명하는 것이 아니라,
**좋은지 나쁜지를 정직한 숫자로 알아내는 것**이다.
결과가 나쁘면 전략을 수정하거나 폐기한다. 그것도 정상적인 결과다.

---

## 2. 데이터

| 항목 | 내용 |
| --- | --- |
| 소스 | 바이낸스 공개 데이터 (data.binance.vision) |
| 심볼 | BTCUSDT USDT-M 선물 |
| 타임프레임 | 15분봉(메인) + 1시간봉(상위 필터) |
| 기간 | 최소 1~2년치. 상승장·하락장·횡보장이 모두 포함되도록 |
| 형식 | OHLCV CSV (open, high, low, close, volume, timestamp) |

**주의:** 데이터 기간이 상승장만 포함하면 결과가 왜곡된다.
반드시 다양한 시장 국면이 섞인 기간을 쓴다.

---

## 3. 지표 계산

설계서 2절의 지표를 계산한다. 표준 정의를 사용한다.

| 지표 | 용도 | 비고 |
| --- | --- | --- |
| ADX | 15분봉 추세 강도 | 표준 기간(보통 14) |
| CCI | 15분봉 진입 트리거 | 표준 기간(보통 20), 기준선 ±100 |
| RSI | 15분봉 방향 확인 | 표준 기간(보통 14) |
| EMA50 | 1시간봉 상위 필터 | 1시간봉 종가 기준 |

라이브러리는 `pandas` + `pandas-ta`(또는 동등물) 사용 권장.
지표 기간 값은 백테스트 중 조정 대상이므로 코드에서 쉽게 바꿀 수 있게
파라미터로 분리한다.

---

## 4. 전략 적용 로직

설계서 2.4~2.7절의 규칙을 그대로 코드로 옮긴다.

```
각 15분봉 캔들 마감 시점마다:
  1. 1시간봉 EMA50 필터 방향 확인
  2. 보유 포지션 있으면:
       - 손절가 도달 검사 (실거래는 거래소 STOP_MARKET, 백테스트는
         캔들 고저가가 손절가를 관통했는지로 시뮬레이션)
       - 익절 조건(CCI 0선 반대돌파 / 부분익절 1R·2R) 검사 (봇 감시 방식)
       - 반대 신호 검사
  3. 보유 포지션 없으면:
       - 롱/숏 진입 6개 조건 검사
       - 중복 진입 방지 조건 검사
       - 충족 시 가상 진입 + 손절가를 함께 기록 (거래소 손절 주문에 해당)
  4. 거래 발생 시 손익 기록
```

**손절 시뮬레이션 주의:** 실거래에서 손절은 거래소 STOP_MARKET 주문이다.
백테스트에서는 캔들의 고가/저가가 손절가를 지나갔다면 그 캔들에서 손절
체결된 것으로 처리한다. 손절가와 익절 조건이 같은 캔들에서 동시에 닿으면,
보수적으로 **손절이 먼저 체결된 것으로 가정**한다(실제 순서를 알 수 없으므로).

## 5. 비용 모델 — 반드시 포함

비용을 빼면 결과가 가짜가 된다. 아래를 모두 반영한다.

| 비용 | 처리 |
| --- | --- |
| 거래 수수료 | 바이낸스 선물 시장가(taker) 수수료율 적용. 진입·청산 양쪽 모두 |
| 슬리피지 | 시장가 체결 시 불리한 방향으로 일정폭 가산 (보수적으로) |
| 펀딩비 | 포지션을 8시간 펀딩 시점 넘겨 보유 시 펀딩비 반영 |

수수료율·슬리피지폭은 코드 파라미터로 분리해, 보수적 값으로 먼저 돌린다.

---

## 6. 반드시 피해야 할 함정

### 6.1 미래 데이터 훔쳐보기 (look-ahead bias)

각 시점에서 **그 시점에 실제로 알 수 있었던 정보만** 사용한다.
- 진행 중인 캔들의 종가를 미리 보지 않는다 (마감된 캔들만).
- 지표 계산에 미래 캔들이 섞이지 않게 한다.
- 신호 발생 캔들의 다음 캔들 시가로 진입하는 것이 현실적.

### 6.2 과최적화 (overfitting)

- 데이터를 둘로 나눈다: 앞 70%(in-sample) / 뒤 30%(out-of-sample).
- 파라미터 조정은 in-sample에서만 한다.
- out-of-sample은 조정이 끝난 뒤 **딱 한 번** 검증용으로 쓴다.
- in-sample만 좋고 out-of-sample에서 무너지면 과최적화된 것이다.

### 6.3 생존자 편향·체리피킹

- 특정 좋은 구간만 골라 보지 않는다. 전체 기간을 본다.
- 결과가 나쁜 구간도 그대로 보고한다.

---

## 7. 결과 지표 — 무엇을 볼 것인가

수익률 하나만 보지 않는다. 아래를 모두 출력한다.

| 지표 | 의미 |
| --- | --- |
| 총 수익률 | 기간 전체 손익 (%) |
| 최대 낙폭(MDD) | 고점 대비 가장 크게 빠진 폭. **가장 중요** |
| 승률 | 전체 거래 중 이긴 비율 |
| 손익비 | 평균 수익 ÷ 평균 손실 |
| 총 거래 횟수 | 너무 적으면 표본 부족, 너무 많으면 수수료 과다 |
| 수수료 총액 | 비용이 수익을 얼마나 갉아먹었는지 |
| 연속 손실 최대 횟수 | 실거래 시 멘탈·청산 위험 가늠 |

**판단 기준:** 총 수익률이 높아도 MDD가 감당 못 할 수준이면(예: -50% 이상)
실거래에서는 청산되거나 봇을 꺼버리게 된다. 수익률과 MDD를 함께 본다.

## 8. 백테스트 진행 흐름

```
1. BTCUSDT 15분봉 + 1시간봉 과거 데이터 다운로드 (1~2년치)
2. 백테스트 코드 작성 (지표 계산 + 전략 규칙 + 비용 모델)
3. 미래 훔쳐보기 차단 검증 (코드 리뷰)
4. in-sample(앞 70%)로 실행, 결과 지표 확인
5. 결과가 부실하면 → 전략 규칙/파라미터 수정 → 4번 반복
6. 만족스러우면 → out-of-sample(뒤 30%)로 단 한 번 검증
7. out-of-sample에서도 통과 → 전략 v1.0 확정
   out-of-sample에서 무너짐 → 과최적화. 전략 재검토
```

## 9. 백테스트 통과 후에도

백테스트 통과는 "과거에 먹혔다"는 뜻이지 "미래에 먹힌다"는 보장이 아니다.
백테스트 v1.0 확정 후에도:

1. 봇 개발 완료 → **바이낸스 테스트넷**(가짜 돈)에서 충분히 가동
2. 테스트넷 결과가 백테스트와 크게 다르면 원인 분석 (체결 지연, 슬리피지 등)
3. 실거래는 **잃어도 괜찮은 아주 작은 금액**으로 시작
4. 점진적으로만 확대

**경고:** 어떤 백테스트 결과도 미래 수익을 보장하지 않는다.
자동매매는 시장 리스크로 손실이 발생할 수 있으며, 투입 자금 전액을
잃을 가능성을 항상 염두에 둔다.
