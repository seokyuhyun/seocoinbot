# 배포·운영 가이드 — 오라클 클라우드 + 바이낸스 선물 봇

> 이 문서는 봇이 완성된 후 **실거래 배포 단계**에서 사용하는 체크리스트다.
> 백테스트·테스트넷 검증이 끝나기 전에는 이 단계로 넘어가지 않는다.
> 클라우드 무료 티어 정책은 변동될 수 있으므로, 실제 진행 시 현행 조건을
> 직접 확인한다.

---

## 0. 선행 조건 (이게 끝나야 배포 시작)

- [ ] 백테스트로 전략 v1.0 확정
- [ ] 9개 모듈 봇 개발 완료
- [ ] 바이낸스 **테스트넷**에서 충분히 가동 검증
- [ ] 테스트넷 결과가 백테스트와 크게 어긋나지 않음 확인

위가 안 끝났으면 아래로 내려가지 않는다.

---

## 1. 오라클 클라우드 VM 준비

- [ ] 오라클 클라우드 계정 생성 (가입 시 무료 티어 현행 조건 확인)
- [ ] 무료 티어(Always Free) 범위의 VM 인스턴스 생성
      - BTC 단일 심볼 봇은 가벼움. 작은 사양으로 충분
      - OS는 리눅스 (Ubuntu LTS 등)
- [ ] **예약된 공인 IP(Reserved Public IP) 할당** ← 핵심
      - 임시(ephemeral) IP가 아니라 예약 IP를 VM에 붙인다
      - 재부팅·중지/시작에도 IP가 바뀌지 않아야 함
      - 무료 티어에 예약 IP가 포함되는지 가입 시 확인
- [ ] 이 예약 IP 주소를 기록해 둔다 (바이낸스 화이트리스트에 쓸 값)

**순서 주의:** VM 생성 → 예약 IP 할당 → (그 다음에) 바이낸스 화이트리스트 등록.
임시 IP 상태로 화이트리스트를 걸면 IP가 바뀌어 봇이 거래소 접속 불가가 된다.

---

## 2. 서버 보안 설정

거래 권한 API 키가 올라가는 서버다. 보안을 느슨하게 두지 않는다.

- [ ] SSH 접속은 **키 인증만** 허용, 비밀번호 로그인 비활성화
- [ ] 방화벽(보안 그룹/iptables)으로 필요한 포트만 개방
      - 아웃바운드: 바이낸스 API 접근용
      - 인바운드: SSH(가능하면 본인 IP로 제한) 외에는 최소화
- [ ] OS 자동 보안 업데이트 설정
- [ ] root 직접 로그인 금지, 일반 사용자 + sudo

---

## 3. 바이낸스 API 키 설정

- [ ] 바이낸스에서 선물 거래용 API 키 발급
- [ ] **출금 권한 비활성화** (거래 + 조회만 허용)
- [ ] **IP 화이트리스트에 1번에서 확보한 예약 공인 IP 등록**
- [ ] 키 종류 분리 (설계서 5장 안전 원칙):
      - 실행 모듈용 거래 키
      - 킬스위치용 독립 거래 키 (실행 모듈 키와 별개)
      - 조회 전용 키 (시장데이터·동기화 등)
- [ ] 모든 키는 서버에서 `.env` 파일 등으로 관리, 코드/레포에 박지 않음
- [ ] `.gitignore` 에 키 파일·`.env` 포함 확인

---

## 4. 봇 배포

- [ ] 레포를 서버로 가져오기 (git clone 등)
- [ ] Python 환경 구성 (가상환경 권장), 의존성 설치
- [ ] `.env` / 설정 파일을 서버에 안전하게 배치 (레포와 별도로 전송)
- [ ] 거래 규격(tick/step 등)은 부팅 시 exchangeInfo API로 받음 — 하드코딩 없음
- [ ] **먼저 테스트넷 설정으로 서버에서 한 번 더 가동 확인**
- [ ] 메인 봇 프로세스 + 킬스위치 독립 프로세스 둘 다 기동되는지 확인
- [ ] 프로세스 관리자 설정 (systemd 등) — 비정상 종료 시 자동 재시작
      - 단, 킬스위치 발동으로 인한 정지는 자동 재시작하면 안 됨.
        설계서 9장의 "발동 후 수동 복구" 원칙과 충돌하지 않게 구성

---

## 5. 가동 후 모니터링

- [ ] 모니터링/알림 모듈의 텔레그램·디스코드 알림이 서버에서 정상 동작
- [ ] 로그가 서버에 정상 기록되는지 확인
- [ ] 부팅 시퀀스 확인: 재시작 시 포지션 동기화가 가장 먼저 실행되고,
      동기화 완료 전 거래가 금지되는지
- [ ] WebSocket 연결 끊김 → 자동 재연결 → stale 시 HALT 동작 확인
- [ ] 거래소 측 손절 주문(STOP_MARKET)이 진입과 함께 실제로 등록되는지 확인
- [ ] 일일 손실 한도·연속 손절 한도 도달 시 거래 중단되는지 확인

---

## 6. 실거래 시작

- [ ] **잃어도 괜찮은 아주 작은 금액**으로 시작
- [ ] 초기에는 자주 로그·알림·포지션을 직접 확인
- [ ] 며칠~몇 주 안정적으로 동작하는 것을 확인한 뒤에만 금액을 점진적 확대
- [ ] 백테스트·테스트넷 대비 실거래 결과가 크게 다르면 즉시 중단하고 원인 분석

---

## 운영 중 주의사항

- 클라우드 무료 티어 정책 변경 가능성 — 주기적으로 계정 상태 확인
- 예약 공인 IP가 유지되는지 확인 (IP가 바뀌면 화이트리스트에서 막힘)
- 거래소 API 변경·점검 공지 모니터링
- 봇 코드 수정 후 배포 시: 테스트넷 재검증 → 실거래 재배포 순서 유지

---

## 경고

자동매매는 설계·배포가 완벽해도 시장 리스크로 손실이 발생할 수 있으며,
투입 자금 전액을 잃을 가능성이 있다. 어떤 백테스트·테스트넷 결과도
미래 수익을 보장하지 않는다. 실거래 자금은 잃어도 생활에 지장이 없는
범위로 엄격히 제한한다.
